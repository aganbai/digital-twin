# V2.0 迭代9 需求文档

## 概述

本迭代包含两大方向：**对话体验增强**（思考过程展示、语音恢复、+号多功能按钮、新会话指令触发、会话列表优化）和**教学管理增强**（头像点击查看信息、课程信息发布与推送、老师画像生成、画像隐私保护）。

---

## R1. 思考过程展示（对话透明化）

### 1.1 功能描述

在对话交互过程中，将系统的思考/处理过程实时展示给用户，增强透明度和用户体验。当 LLM 请求后需要补充 RAG 信息、检索记忆等场景再次调用 LLM 时，给用户展示出完整的处理过程。

### 1.2 展示内容

系统对话处理流程中的以下关键步骤需要向用户展示：

| 步骤 | 标识 | 开始提示 | 完成提示 | 详细信息 |
|------|------|---------|---------|---------|
| RAG 检索 | `rag_search` | "🔍 正在检索知识库..." | "✅ 已检索知识库" | "找到 X 条相关内容" |
| 记忆检索 | `memory_recall` | "🧠 正在检索相关记忆..." | "✅ 已检索记忆" | "找到 X 条相关记忆" |
| 工具调用 | `tool_call` | "🔧 正在搜索增强..." | "✅ 搜索完成" | "使用了 X 工具" |
| LLM 思考 | `llm_thinking` | "💭 AI 正在思考..." | "✅ 生成完成" | 耗时信息 |

### 1.3 展示形式

- **位置**：在 AI 回复消息气泡上方独立展示
- **默认状态**：默认展开显示全部过程
- **可折叠**：用户可手动折叠/展开
- **耗时展示**：每个步骤显示耗时（如 "245ms"）
- **详细信息**：精简版，仅展示数量（如 "找到 3 条相关内容"）

### 1.4 技术要求

- **不持久化**：思考过程仅在对话过程中实时展示，不存储到数据库，刷新后消失
- **SSE 事件**：通过新增 `thinking_step` 类型的 SSE 事件实时推送
- **向后兼容**：新增事件类型不影响旧版客户端（旧版忽略未知事件即可）

### 1.5 SSE 事件流示例

```
event: message
data: {"type":"thinking_step","step":"rag_search","status":"start","message":"🔍 正在检索知识库...","timestamp":1712123456789}

event: message
data: {"type":"thinking_step","step":"rag_search","status":"done","message":"✅ 已检索知识库","detail":"找到 3 条相关内容","duration_ms":245,"timestamp":1712123456890}

event: message
data: {"type":"thinking_step","step":"memory_recall","status":"start","message":"🧠 正在检索相关记忆...","timestamp":1712123456891}

event: message
data: {"type":"thinking_step","step":"memory_recall","status":"done","message":"✅ 已检索记忆","detail":"找到 2 条相关记忆","duration_ms":120,"timestamp":1712123456999}

event: message
data: {"type":"thinking_step","step":"llm_thinking","status":"start","message":"💭 AI 正在思考...","timestamp":1712123457000}

event: message
data: {"type":"delta","content":"你好"}

event: message
data: {"type":"delta","content":"！"}

event: message
data: {"type":"done","conversation_id":123}
```

## R2. 语音功能恢复

### 2.1 功能描述

恢复聊天页面的语音输入功能，完全对齐微信聊天页面的交互体验。

### 2.2 输入栏布局

```
[🔊语音] [输入框] [😊Emoji] [+号]
```

- **左侧**：语音图标（🔊喇叭图标）
- **中间**：文字输入框
- **右侧**：Emoji 表情按钮 + `+` 号按钮
- **发送按钮**：输入框有内容时，`+` 号替换为发送按钮

### 2.3 语音交互流程

1. **默认状态**：显示文字输入框
2. **点击语音图标**：输入框变为「按住说话」按钮，左侧语音图标变为键盘图标
3. **点击键盘图标**：切换回文字输入框，左侧恢复为语音图标
4. **按住说话**：长按录音，松开识别转文字并填入输入框
5. **完全对齐微信聊天页面的语音交互**

## R3. +号多功能按钮（替代新会话图标）

### 3.1 功能描述

剔除原有的新会话图标，改为 `+` 号标识。点击后展开功能面板，支持上传文件/图片/视频和拍照。

### 3.2 +号面板功能项

点击 `+` 号后展开面板，按以下顺序排列：

| 序号 | 功能 | 图标 | 说明 |
|------|------|------|------|
| 1 | 文件 | 📁 | 选择文件（PDF/DOCX/TXT 等） |
| 2 | 相册 | 🖼️ | 从相册选择图片或视频 |
| 3 | 拍摄 | 📷 | 拉起摄像头拍照 |

### 3.3 新会话触发方式

- **剔除**原有的新会话图标/按钮
- 新会话**仅通过用户聊天指令触发**：用户发送特定关键词（如"新话题"、"开始新对话"、"新会话"），后端识别后创建新 session
- **使用场景**：用户觉得当前上下文污染太严重，开一个新会话更清爽

### 4.1 学生点击老师头像

学生在聊天页面点击老师头像后，弹出/跳转到**班级信息页**，展示：

> 注意：不展示老师的教学风格标签（暂不需要）。不展示课程进度信息（暂不需要）。

### 4.2 老师点击学生头像

老师在聊天页面点击学生头像后，弹出/跳转到**学生信息页**，展示：

| 字段 | 来源 | 说明 |
|------|------|------|
| 学生昵称 | 微信昵称 | 学生的微信昵称 |
| 年龄 | 审批时填写 | 学生年龄 |
| 性别 | 审批时填写 | 学生性别 |
| 家庭情况 | 审批时填写 | 家庭背景信息 |
| 老师评语 | 老师填写 | 老师对学生的评价/备注 |

**评语功能**：
- 复用现有的 `class_members.teacher_evaluation` 字段
- 只保留**最新一条**评语（覆盖更新）
- 老师可以直接在学生信息页**增加或修改评语**
- 评语用于对话时的 Prompt 注入，帮助 AI 更好地了解学生

## R5. 课程信息发布

### 5.1 功能描述

允许老师发布课程信息（课件、进度等），发布时相当于**补充知识库**。发布时提醒老师是否需要推送给学生。

### 5.2 课程信息存储

课程信息直接存入 `knowledge_items` 表，使用新的 `item_type = 'course'` 标识：

| 字段 | 说明 |
|------|------|
| item_type | `'course'`（新增类型） |
| title | 课程标题 |
| content | 课程内容（课件文字、进度说明等） |
| scope | 班级维度（`'class'`） |
| scope_id | 班级 ID |

### 5.3 课程管理页面

提供一个简单的专属页面，包含：
- **发布课程**：标题 + 内容 + 选择班级 + 是否推送给学生
- **课程列表**：按时间倒序展示已发布的课程
- **编辑/删除**：支持修改和删除已发布的课程

### 5.4 推送机制

发布课程时，老师可选择是否推送给学生：

1. **应用内推送**：推送通知在每个学生的**聊天页出现**（`teacher_push` 类型消息）
2. **微信订阅消息推送**：
   - 采用**用户主动订阅**方式（`wx.requestSubscribeMessage`）
   - 如果用户没有订阅，在聊天页同步提醒用户开启订阅
   - 需要在微信后台配置消息模板
   - 后端调用微信订阅消息 API 发送

### 5.5 推送内容

**应用内推送消息格式**：
```json
{
  "type": "teacher_push",
  "title": "课程更新通知",
  "content": "曹老师发布了新课程：《色彩基础》",
  "course_id": 123,
  "class_id": 456,
  "created_at": "2026-04-03T10:00:00Z"
}
```

**微信订阅消息模板**：
- 模板标题：课程更新通知
- 关键词：课程名称、老师名称、发布时间

## R6. 老师画像生成

### 6.1 功能描述

系统针对老师的行为等，**定期生成老师画像**，存储在 `users.profile_snapshot` 字段中。

### 6.2 数据来源

| 维度 | 数据来源 | 说明 |
|------|---------|------|
| 教学风格 | 回复频率、回复长度、使用的教学方法 | 分析老师的对话记录 |
| 课程发布频率 | knowledge_items 中 item_type='course' 的记录 | 统计课程发布行为 |
| 学生互动数据 | 对话记录、活跃学生数、平均响应时间 | 分析师生互动情况 |
| 知识库内容特征 | knowledge_items 的内容分析 | 分析教学内容偏好 |

### 6.3 更新频率

与学生画像一致：**每天凌晨定时任务**（由 `MemorySummarizer` 统一调度）。

### 6.4 存储格式

复用现有的 `users.profile_snapshot` 字段（JSON 格式），老师画像的 JSON Schema：

```json
{
  "teaching_style": "苏格拉底式引导",
  "response_frequency": "高频（日均回复50+条）",
  "avg_response_length": "中等（100-200字）",
  "course_publish_frequency": "每周2-3次",
  "active_student_count": 25,
  "knowledge_content_tags": ["美术", "色彩", "素描"],
  "last_updated": "2026-04-03T02:00:00Z"
}
```

## R7. 画像隐私保护

> 注意：R8（会话列表优化）在本文档末尾的功能需求部分。

### 7.1 功能描述

系统生成的学生画像和老师画像，**严禁展示给任何用户**。

### 7.2 保护规则

| 规则 | 说明 |
|------|------|
| 学生画像 | 不展示给学生本人，不展示给老师 |
| 老师画像 | 不展示给老师本人，不展示给学生 |
| 内部使用 | 画像仅用于系统内部（Prompt 注入、个性化推荐等） |
| API 过滤 | 所有返回用户信息的 API，必须过滤掉 `profile_snapshot` 字段 |

### 7.3 技术实现

- 所有返回 `User` 对象的 API 响应中，`profile_snapshot` 字段设为空或不返回
- 前端不展示任何画像相关的 UI 元素
- 画像数据仅在后端 Prompt 构建时读取使用

## R8. 会话列表优化（按会话划分聊天记录）

### 8.1 功能描述

在聊天列表页面中，将每个老师下的聊天记录**按会话（session）分组展示**，允许用户点击某个会话继续对话。解决当前所有会话消息混在一起、用户无法区分和切换会话的问题。

### 8.2 聊天列表展示结构

采用**二级展开**结构，每个老师下展示其所有会话：

```
聊天列表
├── 曹老师（美术）
│   ├── 💬 色彩基础讨论          3月28日   ← 最新会话（无标题总结，显示最后一条消息摘要）
│   ├── 📝 素描技法问答          3月25日   ← 历史会话（有 LLM 总结的标题）
│   └── 📝 期中作业辅导          3月20日   ← 历史会话
├── 李老师（数学）
│   ├── 💬 最近：函数图像分析     3月27日   ← 最新会话
│   └── 📝 方程式练习            3月22日   ← 历史会话
```

**展示规则**：
- 默认每个老师只展示**最新 1 个会话**（即当前活跃会话），可点击展开查看历史会话
- **最新会话**：不需要标题总结，直接显示最后一条消息的摘要（截取前 20 字）
- **历史会话**：显示 LLM 总结的标题
- 每个会话项显示：标题/摘要 + 最后消息时间 + 消息条数
- 点击某个会话 → 进入该会话的聊天详情页（带 `session_id` 参数）

### 8.3 会话标题生成

#### 生成时机

会话标题**在用户创建新会话时**，对**上一个会话**执行标题总结。即：

1. 用户在聊天列表页点击「新会话」→ 后端为上一个活跃会话生成标题
2. 用户在聊天窗口通过指令创建新会话 → 后端为上一个活跃会话生成标题

#### 生成方式

- 调用 LLM 对该会话的对话内容进行总结，生成一个**简短标题**（10-20 字）
- 异步执行，不阻塞新会话的创建流程
- 如果会话消息过少（如 ≤ 2 条），则使用首条用户消息的前 20 字作为标题，不调用 LLM

#### 存储

在 `conversations` 表所关联的会话维度增加标题字段，具体方案：
- 新增 `session_titles` 表：`id`, `session_id`（唯一索引）, `student_persona_id`, `teacher_persona_id`, `title`, `created_at`, `updated_at`

#### LLM Prompt 示例

```
请为以下对话生成一个简短的标题（10-20字），概括对话的主要话题：

{对话内容摘要，取最近10条消息}

要求：
- 标题简洁明了
- 体现对话的核心话题
- 不超过20个字
```

### 8.4 创建新会话的两种方式

#### 方式一：聊天列表页创建

- 在每个老师的会话列表区域，提供「+ 新会话」按钮
- 点击后直接创建新 session 并跳转到聊天详情页
- 同时触发上一个会话的标题总结（异步）

#### 方式二：聊天窗口指令创建

- 复用 R3 的指令触发机制：用户发送 "新话题"、"开始新对话"、"新会话" 等关键词
- 后端识别后创建新 session，返回新的 `session_id`
- 前端收到后切换到新会话，聊天页面显示隔离线
- 同时触发上一个会话的标题总结（异步）

### 8.5 后端 API

#### 获取某老师下的会话列表

复用并增强现有 `GET /api/conversations/sessions` 接口，增加 `teacher_persona_id` 过滤参数：

```
GET /api/conversations/sessions?teacher_persona_id=123&page=1&page_size=20
```

**响应**：
```json
{
  "code": 0,
  "data": {
    "items": [
      {
        "session_id": "sess_abc123",
        "title": null,
        "last_message": "老师，色彩搭配有什么技巧？",
        "last_message_role": "user",
        "message_count": 12,
        "is_active": true,
        "updated_at": "2026-03-28T15:30:00Z"
      },
      {
        "session_id": "sess_def456",
        "title": "素描技法问答",
        "last_message": "谢谢老师的指导！",
        "last_message_role": "user",
        "message_count": 8,
        "is_active": false,
        "updated_at": "2026-03-25T10:00:00Z"
      }
    ],
    "total": 3
  }
}
```

- `title` 为 `null` 表示最新会话（无标题总结），前端显示 `last_message` 摘要
- `is_active` 为 `true` 表示当前活跃会话

### 8.6 前端交互细节

- **展开/收起**：每个老师默认只显示最新会话，点击「查看历史会话 (N)」展开全部
- **会话图标区分**：最新会话用 💬，历史会话用 📝
- **点击跳转**：点击会话项 → `navigateTo('/pages/chat/index?teacher_id=X&session_id=Y&teacher_name=Z')`
- **新会话按钮**：每个老师区域底部显示「+ 新会话」按钮
- **空状态**：如果某老师下没有任何会话，显示「开始第一次对话」引导
