# 冒烟测试用例计划（Smoke Test Plan）

> **维护规则**：每个迭代在 Phase 0（环境准备）阶段，主 Agent 必须根据最新需求文档重新审视并更新本文档，确保冒烟测试用例覆盖全面。
> **自动化工具**：miniprogram-automator SDK（`npm i miniprogram-automator --save-dev`）+ 微信开发者工具模拟器
> **执行时机**：集成测试 + 代码&架构 Review 全部通过后，通过 miniprogram-automator SDK 控制微信开发者工具模拟器自动化执行
> **用例设计原则**：按功能模块划分，每条用例覆盖一个完整的功能场景，验证前后端全链路打通

## 6. 执行依赖关系

```
SM-A02（教师注册）→ SM-G01（创建班级）→ SM-H01（生成分享码）
                                                    ↓
SM-A01（学生注册）→ SM-H01（学生加入）→ SM-D01（学生对话）→ SM-D02（教师真人介入）
                                                    ↓
                                              SM-H02（定向邀请）→ SM-Q01（非目标学生扫码）

--- 迭代6 新增依赖链 ---
SM-D01（学生对话）→ SM-N05（记忆自动分层）→ SM-N01（教师查看记忆）→ SM-N02/N03/N04（编辑/删除/合并）
SM-A02（教师注册）→ SM-O01（设置教学风格）→ SM-D01（验证风格生效）
SM-H01（生成分享码）→ SM-P01（生成二维码）→ SM-P02（保存二维码）
SM-A02（教师注册）→ SM-R01（导入聊天记录）
SM-A01/A02（注册）→ SM-S01/S02（TabBar 布局验证）
SM-T01（Docker 部署）← 独立执行，不依赖其他用例
```

> ℹ️ **说明**：
> - SM-A01/SM-A02 是所有用例的前置条件，必须最先执行
> - SM-D02（教师真人介入）依赖 SM-D01（学生对话）产生的对话数据
> - SM-N01~N04（记忆管理）依赖 SM-N05 或 SM-D01 产生的记忆数据
> - SM-T01（Docker 部署）独立执行，可与其他用例并行
> - 其余用例相对独立，可并行执行
> - 每条用例由 ci_test_agent 编排的独立 sub agent 执行，失败后定位问题归属，反馈给对应开发 Agent 修复
>
> ⚠️ **并行执行与非阻塞修复规则**：
> - ci_test_agent 先分析用例依赖图，将用例分为串行链和独立用例
> - 有数据依赖的用例按依赖顺序串行执行，无数据依赖的用例尽量并行执行
> - 某条用例失败进入修复循环时，**不阻塞无依赖的其他用例继续执行**，仅阻塞依赖该用例数据的后续用例
> - 修复完成后，重新执行失败用例及其依赖链上的后续用例
